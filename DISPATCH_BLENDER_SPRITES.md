# Blender-Rendered Tree Sprites: Replace Procedural Generation with Atlas

## Context

Tree sprites are currently generated procedurally in `src/sprites.rs` at 16×16 resolution.
We have Blender-rendered conifer sprites at 256×256 in `assets/sprites/conifer_atlas_256.bin`
(8 variants, packed as `R | (G<<8) | (B<<16) | (HEIGHT<<24)` — same format as current).

The atlas was rendered from a 3D conifer model in Blender at 45° orthographic with:
- Flat albedo only (no baked lighting — the raytrace shader does all lighting)
- Vertex-color noise texture for organic foliage detail
- Height encoded in alpha for shadow casting
- 8 variants with randomized foliage rotation/scale

This is a Trunk/WASM project — no filesystem access at runtime.
Assets are embedded at compile time via `include_bytes!` (same pattern as the
`include_str!` used for `.toml` config files throughout the codebase).

## Changes Required (2 files, 2 changes)

### 1. `src/sprites.rs` — Replace entire file

Replace the entire contents of `src/sprites.rs` with:

```rust
//! Tree sprite atlas — loads pre-rendered Blender sprites embedded at compile time.
//!
//! The atlas binary is generated by:
//!   blender --background assets/blender/conifer.blend --python assets/blender/render_conifer.py
//!
//! Format: R | (G << 8) | (B << 16) | (HEIGHT << 24) per pixel, row-major, variant-major.
//! Index:  sprites[variant * SPRITE_SIZE^2 + y * SPRITE_SIZE + x]

pub const SPRITE_SIZE: u32 = 256;
pub const SPRITE_VARIANTS: u32 = 8;

/// Embedded atlas binary (2 MB, 8 variants × 256×256 × u32).
static ATLAS: &[u8] = include_bytes!("../assets/sprites/conifer_atlas_256.bin");

pub fn generate_tree_sprites() -> Vec<u32> {
    let total = (SPRITE_SIZE * SPRITE_SIZE * SPRITE_VARIANTS) as usize;
    let expected_bytes = total * 4;

    assert_eq!(
        ATLAS.len(),
        expected_bytes,
        "Sprite atlas size mismatch: expected {} bytes, got {}. Re-run render_conifer.py.",
        expected_bytes,
        ATLAS.len()
    );

    let mut data = vec![0u32; total];
    for (i, chunk) in ATLAS.chunks_exact(4).enumerate() {
        data[i] = u32::from_le_bytes([chunk[0], chunk[1], chunk[2], chunk[3]]);
    }
    data
}
```

## What does NOT need to change

- **`src/gpu_init.rs`**: Buffer is sized dynamically from `sprite_data.len()` — automatically grows from 8 KB to 2 MB.
- **`src/main.rs`**: Already imports `sprites::generate_tree_sprites` — function signature unchanged.
- **`render_tree()` in `raytrace.wgsl`**: Already uses `SPRITE_SIZE` constant for UV→pixel mapping — works at any resolution.
- **Shadow map shader**: Doesn't reference `SPRITE_SIZE` — no change needed.
- **`Trunk.toml`**: No change needed — `include_bytes!` embeds at compile time, not served at runtime.

## How it works in the shader

From `raytrace.wgsl` `render_tree()` (lines ~2032-2085):
1. Per-tree `id_hash` selects a variant (0-7) and a random rotation angle
2. `sample_sprite(variant, u, v)` indexes into the atlas → returns `vec4(r,g,b, height)`
3. Transparent pixels (height=0) show ground underneath
4. Trunk pixels (height < 0.4) are darkened and rendered at ground level
5. Canopy pixels get full color, tinted ±15% by `id_hash` for per-tree variation
6. The main render loop then applies: `albedo * (ambient + sun_color * shadow * 0.85) + proximity_glow`

The sprite only carries flat albedo + height. All lighting is dynamic.

## Testing

After making the changes, `cargo build` (or `trunk serve`) should compile and the trees
should appear as textured 3D-rendered conifers instead of the previous procedural circles/diamonds.

If the atlas file is missing or wrong size, the build will fail with a clear assert message
telling you to re-run the Blender render script.
